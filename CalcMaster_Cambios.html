Resumen ejecutivo

Se realizó una revisión completa del proyecto CalcMaster. Se identificaron y corrigieron 6 bugs, se eliminaron 2 bloques de código muerto, se aplicaron 2 correcciones de seguridad y se implementaron 5 mejoras funcionales de UX. A continuación el detalle completo por archivo.

Bugs corregidos

BUG FIX #1
División por cero retornaba NaN

Problema: Al dividir entre 0, el display mostraba "NaN" — un texto técnico sin sentido para el usuario.
Solución: Ahora detecta el caso val2 == 0.0, pone result = null y muestra DIV/0 en rojo. Las operaciones quedan bloqueadas hasta presionar C.

BUG FIX #2
Resultado con decimales excesivos / imprecisos

Problema: Se usaba "%.2f".format(result) fijo, por lo que 1 / 2 = 0.50 en vez de 0.5.
Solución: Se usa "%.6f".format(result).trimEnd('0').trimEnd('.') — elimina ceros innecesarios y muestra solo la precisión necesaria.

BUG FIX #3
Overflow silencioso del display

Problema: No había límite en la longitud del número ingresado. Números de 20+ dígitos desbordaban el display sin aviso.
Solución: Límite estricto de 15 caracteres en la entrada numérica.

BUG FIX #4
Estado de error bloqueaba la aplicación

Problema: Tras un error (NaN), la app quedaba en estado corrupto — presionar números seguía acumulando sobre el error.
Solución: El flag isError detecta el estado; al presionar cualquier dígito, limpia el error y comienza fresco. Los operadores están deshabilitados en modo error.

BUG FIX #5
No había botón de punto decimal

Problema: La calculadora no permitía ingresar números decimales como 3.14 — solo podía mostrarlos como resultado.
Solución: Se agrega el botón . con lógica que previene duplicación (si ya hay un punto, no agrega otro).

BUG FIX #6
No había forma de corregir un dígito sin limpiar todo

Problema: Si el usuario escribía 1234 y quería borrar el 4, debía presionar C y empezar desde cero.
Solución: Se agrega el botón ⌫ que borra el último carácter. Si queda vacío, retorna a "0".

Correcciones de seguridad

SEGURIDAD #1
Vault expuesto en backup automático de Android

Problema: Los datos del HELIOS TERMINAL se guardaban en SharedPreferences (helios_vault). Con allowBackup=true y reglas vacías, Android sincronizaba esos datos sin cifrar a Google Drive automáticamente.
Solución: Se configuraron explícitamente backup_rules.xml y data_extraction_rules.xml para excluir helios_vault.xml de tanto cloud backup como device-to-device transfer.

SEGURIDAD #2
Permiso VIBRATE declarado sin uso

Problema: AndroidManifest.xml declaraba android.permission.VIBRATE pero el código nunca lo usa (solo utiliza haptic feedback que no requiere permiso).
Solución: Permiso eliminado. Principio de mínimo privilegio.

Limpieza de código

LIMPIEZA #1
Colores huérfanos del template inicial eliminados

Problema: Color.kt contenía 6 variables del template por defecto de Android Studio (Purple80, PurpleGrey80, Pink80, Purple40, PurpleGrey40, Pink40) que nunca se referenciaban en ningún archivo.
Solución: Eliminadas. Solo se mantienen los 4 colores propios: HeliosGreen, HeliosOrange, HeliosBlack, HeliosDarkGrey.

Mejoras de UX

MEJORA #1
Indicador visual del operador activo

El operador actual (+, -, *, /) ahora aparece en naranja (HeliosOrange) en la esquina superior derecha del display mientras se espera el segundo operando. El usuario siempre sabe qué operación está en curso.

MEJORA #2
Font size adaptativo según longitud del número

El display ajusta automáticamente el tamaño de fuente para evitar que números largos se corten:

• ≤ 8 dígitos → 64sp (tamaño original)
• 9–12 dígitos → 44sp
• > 12 dígitos → 32sp

MEJORA #3
Display de error diferenciado en rojo

Cuando ocurre un error (DIV/0), el texto del display cambia a #FF4444 (rojo) para distinguirlo visualmente de un resultado normal. Al presionar C o cualquier dígito, vuelve al verde normal.

MEJORA #4
Botón C diferenciado en rojo

El botón Clear (C) ahora tiene borde e icono en rojo (#FF4444) mediante el nuevo parámetro isDanger en HeliosButton. Comunica visualmente su naturaleza destructiva sin romper el layout.

MEJORA #5
Resultados enteros con precisión correcta para números grandes

Al mostrar resultados enteros, se cambió result.toInt() por result.toLong() para soportar números grandes (hasta ~9.2 × 10¹⁸) sin overflow en Java/Kotlin.
